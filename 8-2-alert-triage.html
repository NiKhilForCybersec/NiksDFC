Alert Triage Process

L1 Fundamentals L2 Technical 20 min read

This guide provides a structured approach for SOC analysts to triage Microsoft Defender for Cloud security alerts efficiently and effectively. Following this process ensures consistent handling, proper documentation, and appropriate escalation.

Target Audience

This playbook is designed for L1/L2 SOC Analysts handling MDC alerts. L3 analysts and incident responders should reference the specific threat playbooks for detailed investigation procedures.

Step 1: Initial Assessment

When an alert arrives, quickly assess its priority and category.

Severity Response Matrix

Severity Response SLA Owner Actions
High 15 minutes L2/L3 Analyst Immediate triage, consider containment, escalate if needed
Medium 1 hour L1/L2 Analyst Standard triage, investigate, document findings
Low 4 hours L1 Analyst Queue-based handling, batch similar alerts
Informational 24 hours L1 Analyst Review for patterns, close if expected behavior

MDC Alert Categories

Category Common Alert Types Playbook
Virtual Machine Malware, suspicious process, brute force, crypto mining VM Playbook
Identity/Access Impossible travel, suspicious login, privilege escalation Identity Playbook
Storage/Data Anomalous access, data exfiltration, public exposure Data Playbook
Network C2 communication, port scanning, DNS tunneling Network Playbook
Container/K8s Privileged container, exposed dashboard, image vuln Container Guide

Step 2: Context Gathering

Gather essential context before making a verdict.

Information Gathering Checklist

SecurityAlert
| where TimeGenerated > ago(7d)
| where CompromisedEntity contains targetResource 
    or ExtendedProperties contains targetResource
| project 
    TimeGenerated,

    AlertSeverity,

    Tactics
| order by TimeGenerated desc

    CompromisedEntity,
    Tactics

Step 3: Verdict Decision

Based on your investigation, classify the alert.

Verdict Types

True Positive (TP)

Definition: Confirmed malicious activity requiring response.

Action: Escalate to incident response, begin containment.

Benign True Positive (BTP)

Definition: Detected correctly but is expected/authorized activity.

Action: Document, consider exclusion rule, close as resolved.

False Positive (FP)

Definition: Detection was incorrect; no actual threat.

Action: Dismiss alert, report to detection engineering, tune rules.

Undetermined

Definition: Cannot determine with available information.

Action: Escalate to L2/L3, request additional data sources.

Step 4: Action Execution

True Positive Actions

1. Immediate Containment

Isolate affected resource if active threat (network isolation, disable account)

2. Escalate to L3/IR

Create incident ticket, assign to incident response team

3. Preserve Evidence

Capture VM disk snapshot, export logs, screenshot alerts

4. Notify Stakeholders

Alert system owners, management as per communication plan

5. Document in Ticket

Record timeline, actions taken, evidence locations

False Positive Actions

PowerShell 
# Dismiss alert in MDC via API
$alertId = "your-alert-id"
$subscriptionId = "your-subscription-id"

# Dismiss with reason
$dismissBody = @{
    properties = @{
        status = "Dismissed"

Important

Always document WHY an alert is being dismissed. Include ticket numbers, user verification, and change request references for audit purposes.

  • Follow SLAs: High=15min, Medium=1hr, Low=4hr, Info=24hr

  • Use specific playbooks for different alert categories
  • Document everything - verdicts, reasoning, actions taken
  • When in doubt, escalate to L2/L3 rather than closing prematurely
  • Report false positives to detection engineering for rule tuning
