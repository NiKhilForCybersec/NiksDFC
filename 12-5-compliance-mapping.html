Compliance Framework Mapping

L1 Reference L2 Compliance Reference Guide

Microsoft Defender for Cloud provides built-in compliance assessments against major regulatory frameworks. This reference maps common compliance requirements to MDC recommendations and controls, helping you understand how MDC supports your compliance posture.

Important Disclaimer

MDC compliance assessments are technical controls only. Full regulatory compliance requires organizational policies, procedures, and controls beyond technical implementation. MDC helps with the technical portion but doesn't guarantee compliance.

CIS Azure Foundations Benchmark

The CIS (Center for Internet Security) Azure Foundations Benchmark provides prescriptive guidance for establishing a secure Azure configuration.

CIS Control Requirement MDC Recommendation
1.1 Ensure MFA is enabled for all privileged users MFA should be enabled on accounts with owner permissions
1.3 Ensure guest users are reviewed monthly Guest accounts should be removed from subscription
2.1 Ensure Defender for Cloud is enabled Azure Defender should be enabled
2.11 Ensure ASC default policy is not disabled Security policy should be enabled
4.1 Ensure SQL server auditing is enabled Auditing should be enabled on SQL servers
4.4 Ensure SQL TDE is enabled TDE should be enabled on SQL databases
5.1 Ensure diagnostic logging is enabled Diagnostic logs should be enabled
6.1 Ensure RDP access is restricted Management ports should be protected with JIT
6.2 Ensure SSH access is restricted Management ports should be closed on VMs
7.1 Ensure VM disks are encrypted Virtual machines should encrypt temp disks
8.1 Ensure Key Vault is recoverable Key vaults should have purge protection enabled

NIST SP 800-53 Mapping

NIST Special Publication 800-53 provides security and privacy controls for federal information systems. MDC maps to key control families:

AC - Access Control
AC-2 Account Management Deprecated accounts should be removed, External accounts with owner permissions should be removed
AC-3 Access Enforcement RBAC should be used on Kubernetes Services, Storage accounts should restrict network access
AC-6 Least Privilege A maximum of 3 owners should be designated, Service principals should be used instead of management certificates
AC-17 Remote Access Management ports should be protected with JIT network access control
AU - Audit and Accountability
AU-2 Audit Events Diagnostic logs in Azure resources should be enabled
AU-3 Content of Audit Records SQL servers should have auditing enabled, Key Vault logging should be enabled
AU-6 Audit Review Activity log should be retained for at least 365 days
AU-12 Audit Generation Resource logs should be enabled in Azure services
SC - System and Communications Protection
SC-7 Boundary Protection NSG rules for internet-facing VMs should be hardened, Subnets should be associated with NSG
SC-8 Transmission Confidentiality HTTPS should be enforced on web apps, TLS should be the latest version
SC-12 Cryptographic Key Management Key Vault keys should have an expiration date
SC-28 Protection of Information at Rest TDE on SQL databases should be enabled, Storage account encryption should use CMK
SI - System and Information Integrity
SI-2 Flaw Remediation System updates should be installed, Vulnerabilities should be remediated
SI-3 Malicious Code Protection Endpoint protection should be installed, Defender for Servers should be enabled
SI-4 System Monitoring Microsoft Defender for Cloud should be enabled

PCI DSS v4.0 Mapping

Payment Card Industry Data Security Standard requirements mapped to MDC:

PCI Requirement Description MDC Controls
1.x Install and maintain network security controls NSG recommendations, Firewall rules, Network segmentation
2.x Apply secure configurations Vulnerability assessment, CIS benchmarks, Secure score
3.x Protect stored account data Encryption recommendations, Key Vault, TDE, Storage encryption
4.x Protect cardholder data with strong cryptography TLS recommendations, HTTPS enforcement, Certificate management
5.x Protect systems from malicious software Defender for Servers, Endpoint protection, Malware scanning
6.x Develop and maintain secure systems Vulnerability scanning, Container image scanning, Code scanning
7.x Restrict access by business need to know RBAC recommendations, Least privilege, JIT access
8.x Identify users and authenticate access MFA recommendations, Identity protection, Service account hygiene
10.x Log and monitor all access Diagnostic logging, Activity logs, Security alerts
11.x Test security regularly Continuous assessment, Vulnerability scanning, Attack path analysis

Adding Compliance Standards

  1. Azure Portal ‚Üí Defender for Cloud ‚Üí Regulatory compliance
  2. Click "Manage compliance policies"
  3. Select your subscription
  4. Under "Industry & regulatory standards", click "Add more standards"
  5. Select desired standard and click "Add"
Azure CLI 
# List available standards
az security regulatory-compliance-standards list \
    --query "[].{Name:name,State:state}" -o table

# Assign a standard to subscription
az policy assignment create \
    --name "PCI-DSS-v4" \
    --display-name "PCI DSS v4.0" \
    --policy-set-definition "/providers/Microsoft.Authorization/policySetDefinitions/496eeda9-8f2f-4d5e-8dfd-204f0a92ed43" \
    --scope "/subscriptions/your-subscription-id"
PowerShell 
# Get built-in compliance initiatives
Get-AzPolicySetDefinition | 
    Where-Object { $_.Properties.metadata.category -eq "Regulatory Compliance" } |
    Select-Object Name, @{N="DisplayName";E={$_.Properties.DisplayName}}

# Common initiative IDs:
# CIS Azure 2.0.0: /providers/Microsoft.Authorization/policySetDefinitions/06f19060-9e68-4070-92ca-f15cc126059e
# NIST 800-53 R5: /providers/Microsoft.Authorization/policySetDefinitions/179d1daa-458f-4e47-8086-2a68d0d6c38f
# PCI DSS v4: /providers/Microsoft.Authorization/policySetDefinitions/496eeda9-8f2f-4d5e-8dfd-204f0a92ed43
# ISO 27001: /providers/Microsoft.Authorization/policySetDefinitions/89c6cddc-1c73-4ac1-b19c-54d1a15a42f2

# Assign initiative
New-AzPolicyAssignment `
    -Name "NIST-800-53-R5" `
    -DisplayName "NIST SP 800-53 Rev. 5" `
    -PolicySetDefinition (Get-AzPolicySetDefinition -Id "/providers/Microsoft.Authorization/policySetDefinitions/179d1daa-458f-4e47-8086-2a68d0d6c38f") `
    -Scope "/subscriptions/your-subscription-id"

Exporting Compliance Reports

PowerShell 
# Export compliance state via REST API
$subscriptionId = (Get-AzContext).Subscription.Id
$standard = "Azure-CIS-1.4.0"

$uri = "https://management.azure.com/subscriptions/$subscriptionId/providers/Microsoft.Security/regulatoryComplianceStandards/$standard/regulatoryComplianceControls?api-version=2019-01-01-preview"

$response = Invoke-AzRestMethod -Method GET -Uri $uri
$controls = ($response.Content | ConvertFrom-Json).value

# Export to CSV
$controls | Select-Object @{N="Control";E={$_.name}}, 
    @{N="State";E={$_.properties.state}},
    @{N="PassedAssessments";E={$_.properties.passedAssessments}},
    @{N="FailedAssessments";E={$_.properties.failedAssessments}} |
    Export-Csv -Path "compliance-report.csv" -NoTypeInformation
KQL 
// Compliance state over time
SecureScoreControls
| where TimeGenerated > ago(30d)
| summarize 
    HealthyResources = sum(HealthyResources),
    UnhealthyResources = sum(UnhealthyResources)
    by bin(TimeGenerated, 1d)
| extend CompliancePercent = round(100.0 * HealthyResources / (HealthyResources + UnhealthyResources), 1)
| project TimeGenerated, CompliancePercent
| render timechart
Key Takeaways
  • MDC provides technical control assessment - not full compliance certification
  • MCSB (Microsoft Cloud Security Benchmark) is the default standard
  • Add industry-specific standards (PCI, HIPAA, etc.) based on requirements
  • Compliance standards are implemented as Azure Policy initiatives
  • Export reports regularly for audit evidence
  • Exemptions should be documented with business justification
  • Custom standards can be created for internal policies
