VM Threat Investigation Playbook

L1 SOC L2 SOC Operational Playbook

This playbook provides step-by-step procedures for investigating Defender for Servers alerts on virtual machines. Follow these procedures to ensure consistent, thorough investigations.

Before You Begin

Ensure you have the following access:

  • Azure Portal - Security Reader role (minimum)
  • Microsoft 365 Defender portal access
  • Log Analytics workspace query permissions
  • Sentinel Responder role (for incident management)

Step 1: Initial Triage (5 minutes)

L1 Triage Checklist

Quick Context Queries

KQL 
// Get alert details and related alerts
let vmName = "affected-vm-name";
let alertTime = datetime(2024-01-15T10:30:00Z);
SecurityAlert
| where TimeGenerated between ((alertTime - 1h) .. (alertTime + 1h))
| where CompromisedEntity == vmName or ExtendedProperties has vmName
| project 
    TimeGenerated,
    AlertName,
    AlertSeverity,
    Description,
    Tactics,
    Techniques,
    ExtendedProperties
| order by TimeGenerated desc
KQL 
// Check VM history - alerts in last 30 days
let vmName = "affected-vm-name";
SecurityAlert
| where TimeGenerated > ago(30d)
| where CompromisedEntity == vmName
| summarize 
    TotalAlerts = count(),
    HighSeverity = countif(AlertSeverity == "High"),
    UniqueAlertTypes = dcount(AlertName)
| extend RiskIndicator = iff(TotalAlerts > 10, "High Activity - Review Needed", "Normal")

Quick Decision Matrix

Condition Action
Known false positive (documented) Close as FP, note in ticket
High severity + Production VM Escalate to L2 immediately
Medium severity + Dev/Test VM Continue investigation, standard SLA
Multiple VMs affected Escalate as potential incident
Unsure / Complex Escalate to L2 with notes

Step 2: Enrichment & Context

MDE Device Timeline

Navigate to Microsoft 365 Defender portal â†’ Devices â†’ [VM Name] â†’ Timeline

Pro Tip
Filter timeline to 1 hour before and after alert time. Look for:
  • Process creations (especially from unusual parents)
  • Network connections to external IPs
  • File modifications in sensitive directories
  • Registry changes

Process Tree Analysis

KQL (MDE) 
// Process tree for suspicious activity
let deviceName = "affected-vm-name";
let alertTime = datetime(2024-01-15T10:30:00Z);
DeviceProcessEvents
| where Timestamp between ((alertTime - 30m) .. (alertTime + 30m))
| where DeviceName =~ deviceName
| project 
    Timestamp,
    FileName,
    FolderPath,
    ProcessCommandLine,
    InitiatingProcessFileName,
    InitiatingProcessCommandLine,
    AccountName,
    ProcessId,
    InitiatingProcessId
| order by Timestamp asc

Network Connections

KQL (MDE) 
// External network connections around alert time
let deviceName = "affected-vm-name";
let alertTime = datetime(2024-01-15T10:30:00Z);
DeviceNetworkEvents
| where Timestamp between ((alertTime - 1h) .. (alertTime + 1h))
| where DeviceName =~ deviceName
| where RemoteIPType == "Public"
| project 
    Timestamp,
    RemoteIP,
    RemotePort,
    RemoteUrl,
    InitiatingProcessFileName,
    InitiatingProcessCommandLine
| summarize 
    ConnectionCount = count(),
    Ports = make_set(RemotePort),
    Processes = make_set(InitiatingProcessFileName)
    by RemoteIP
| order by ConnectionCount desc

User Activity Context

KQL 
// Who was logged in during the alert?
let vmName = "affected-vm-name";
let alertTime = datetime(2024-01-15T10:30:00Z);
SecurityEvent
| where TimeGenerated between ((alertTime - 2h) .. (alertTime + 1h))
| where Computer =~ vmName
| where EventID in (4624, 4625, 4648, 4672) // Logon events
| project 
    TimeGenerated,
    EventID,
    AccountName = TargetUserName,
    LogonType,
    IpAddress,
    WorkstationName
| order by TimeGenerated desc

Step 3: Scope the Impact

Check for Lateral Movement

KQL (MDE) 
// Lateral movement indicators from affected VM
let sourceDevice = "affected-vm-name";
let alertTime = datetime(2024-01-15T10:30:00Z);
DeviceNetworkEvents
| where Timestamp between ((alertTime - 1h) .. (alertTime + 4h))
| where DeviceName =~ sourceDevice
| where RemoteIPType == "Private" // Internal network
| where RemotePort in (22, 23, 135, 139, 445, 3389, 5985, 5986) // Admin ports
| summarize 
    ConnectionAttempts = count(),
    FirstSeen = min(Timestamp),
    LastSeen = max(Timestamp)
    by RemoteIP, RemotePort
| order by ConnectionAttempts desc

Check Other Hosts for Same IOCs

KQL (MDE) 
// Search for same malicious file hash across environment
let maliciousHash = "abc123..."; // SHA256 from alert
DeviceFileEvents
| where Timestamp > ago(7d)
| where SHA256 == maliciousHash
| summarize 
    AffectedDevices = dcount(DeviceName),
    DeviceList = make_set(DeviceName),
    FirstSeen = min(Timestamp),
    LastSeen = max(Timestamp)

// Search for same process name/command line
let suspiciousProcess = "mimikatz"; // or command line pattern
DeviceProcessEvents
| where Timestamp > ago(7d)
| where FileName contains suspiciousProcess 
    or ProcessCommandLine contains suspiciousProcess
| summarize 
    Devices = make_set(DeviceName),
    Users = make_set(AccountName)
    by FileName, ProcessCommandLine

Data Access Check

KQL 
// Check for data access from compromised VM
let vmIP = "10.0.1.50"; // IP of affected VM
let alertTime = datetime(2024-01-15T10:30:00Z);
// Storage access
StorageBlobLogs
| where TimeGenerated between ((alertTime - 1h) .. (alertTime + 4h))
| where CallerIpAddress == vmIP
| summarize 
    ReadOps = countif(OperationName == "GetBlob"),
    WriteOps = countif(OperationName == "PutBlob"),
    Containers = make_set(ObjectKey)
    by AccountName

Step 4: Response Actions

âš ï¸ High-Impact Actions

The following actions require L2/L3 approval and change management for production systems:

  • VM isolation
  • Account disabling
  • Network rule changes

Containment Options

Option 1: MDE Network Isolation (Recommended)

Isolates VM while maintaining management access.

PowerShell 
# Isolate via MDE API
$deviceId = "device-id-from-mde"
$isolationComment = "Isolated due to alert ID: XXX"

# Using MDE API
Invoke-MgGraphRequest -Method POST `
    -Uri "https://api.securitycenter.microsoft.com/api/machines/$deviceId/isolate" `
    -Body @{
        Comment = $isolationComment
        IsolationType = "Full" # or "Selective"
    }

Option 2: Azure NSG Isolation

Azure CLI 
# Create isolation NSG and apply to VM NIC
az network nsg create \
    --resource-group "your-rg" \
    --name "NSG-Isolation-vmname"

# Deny all inbound
az network nsg rule create \
    --resource-group "your-rg" \
    --nsg-name "NSG-Isolation-vmname" \
    --name "DenyAllInbound" \
    --priority 100 \
    --direction Inbound \
    --access Deny \
    --source-address-prefixes "*"

# Deny all outbound (except Azure management)
az network nsg rule create \
    --resource-group "your-rg" \
    --nsg-name "NSG-Isolation-vmname" \
    --name "DenyAllOutbound" \
    --priority 100 \
    --direction Outbound \
    --access Deny \
    --destination-address-prefixes "*"

# Apply to NIC
az network nic update \
    --resource-group "your-rg" \
    --name "vm-nic-name" \
    --network-security-group "NSG-Isolation-vmname"

Block Malicious IPs/Domains

PowerShell (MDE) 
# Add IP to MDE block list
$indicator = @{
    indicatorValue = "192.168.1.100"
    indicatorType = "IpAddress"
    action = "Block"
    title = "Blocked due to incident XXX"
    description = "C2 server identified in investigation"
    severity = "High"
    recommendedActions = "Block and investigate any connections"
}

Invoke-MgGraphRequest -Method POST `
    -Uri "https://api.securitycenter.microsoft.com/api/indicators" `
    -Body $indicator

Block File Hash

PowerShell (MDE) 
# Block malicious file hash
$fileIndicator = @{
    indicatorValue = "abc123sha256hash..."
    indicatorType = "FileSha256"
    action = "BlockAndRemediate"
    title = "Malware hash from incident XXX"
    description = "Ransomware dropper"
    severity = "High"
}

Invoke-MgGraphRequest -Method POST `
    -Uri "https://api.securitycenter.microsoft.com/api/indicators" `
    -Body $fileIndicator

Disable Compromised Account

PowerShell (Entra ID) 
# Disable user account
$userId = "[email protected]"

# Disable sign-in
Update-MgUser -UserId $userId -AccountEnabled:$false

# Revoke all sessions
Revoke-MgUserSignInSession -UserId $userId

# Force password reset on re-enable
Update-MgUser -UserId $userId -PasswordProfile @{
    ForceChangePasswordNextSignIn = $true
}

Common VM Alert Types

Suspicious PowerShell Activity

What to check:

  • Full command line - is it encoded (base64)?
  • Parent process - what launched PowerShell?
  • Network connections after execution
  • Files created/modified

Common false positives:

  • SCCM/Intune management scripts
  • Azure Automation runbooks
  • IT admin troubleshooting

True positive indicators:

  • Download cradles (IEX, Invoke-WebRequest to external)
  • Encoded commands with suspicious decoded content
  • AMSI bypass attempts
Ransomware Activity Detected

IMMEDIATE ACTIONS:

  1. Isolate VM immediately (MDE isolation preferred)
  2. Check for lateral movement to other VMs
  3. Identify patient zero and attack vector
  4. Engage incident response team

Do NOT:

  • Shut down VM (lose volatile memory)
  • Delete files (forensic evidence)
  • Pay ransom before consulting legal/IR
Crypto Mining Detected

Investigation focus:

  • How did miner get installed? (vulnerability, stolen creds)
  • What else did attacker do? (miners often secondary payload)
  • Check for persistence mechanisms

Response:

  • Kill mining process
  • Remove persistence
  • Patch initial access vulnerability
  • Rotate any exposed credentials
Brute Force Attack Detected

Check:

  • Were any attempts successful? (EventID 4624 after 4625s)
  • Source IP - is it external or internal?
  • Targeted accounts - service accounts? admins?

Response:

  • Block source IP at NSG/firewall
  • Enable JIT access if not already
  • Reset passwords for targeted accounts if successful
  • Review why RDP/SSH is exposed
Investigation Best Practices
  • Always check MDE timeline for full context - MDC alerts are just the starting point
  • Look for lateral movement before focusing on single host
  • Document everything - screenshots, queries, findings
  • Escalate early if unsure - better to involve L2 than miss something
  • Check for persistence - attackers rarely do just one thing
  • Preserve evidence before remediation - memory dumps, disk images for critical incidents
